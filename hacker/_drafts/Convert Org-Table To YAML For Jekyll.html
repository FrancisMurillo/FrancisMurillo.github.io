Problem

So in making the page for my watch list for the other watcher blog, I have to convert my unofficial list in Org-Table format to a YAML format. There are other default formats such as CSV that Jekyll reads but in keeping with the spirit, I opt to do it as such. The sad thing about this though is that there is no YAML exporter for org-table-export, so I made my own. Here is the snippet to do so. 

(defun fn/orgtbl-to-yaml (table params)
  "Convert an org table to yaml."
  (lexical-let ((headers
       (mapcar
        (lambda (header) ;; Convert them to a proper name
          (replace-regexp-in-string
           " " "-"
           (downcase header)))
        (car table)))
      (zip
       (lambda (xs ys)
         (lexical-let ((ms xs)
             (ns ys)
             (zs nil))
           (while (or (car ms) (car ns))
             (push (cons (car ms) (car ns)) zs)

             (setq ms (cdr ms)
                ns (cdr ns)))

           (reverse zs)))))
    (orgtbl-to-generic
     table
     (org-combine-plists
      (list
       :skip 1 ;; Skip header
       :lfmt
       (lambda (values)
         (concat
          "- "
          (string-join
           (mapcar
            (lambda (pair)
              (lexical-let ((header (car pair))
                  (value (cdr pair)))
                (format "%s: \"%s\"" header value)))
            (funcall zip headers values))
           "\n  "))))
      params))))

And to use it, you point the mark at a table and use org-table-export and at the format selection you type fn/orgtbl-to-yaml. Obviously, you have to export it to your _data Jekyll directory to work as stated by the manual. But of course, we are here to explore further to what it means.

Org-Table Export

Initially I thought I had to create a table parser and transformer but there is usually a command or function that does what you want albeit with a little tweaking. The command org-table-export is our friend here but it is lacking when presented with the format: CSV, TSV, HTML, LaTeX and others. One thing to look at where this option comes from is org-export-backends but not really helpful to configure, if you add to it you need to run this script. 

(progn
  (setq org-export-registered-backends
     (cl-remove-if-not
      (lambda (backend)
        (let ((name (org-export-backend-name backend)))
          (or (memq name val)
             (catch 'parentp
               (dolist (b val)
                 (and (org-export-derived-backend-p b name)
                    (throw 'parentp t)))))))
      org-export-registered-backends))
  (let ((new-list (mapcar #'org-export-backend-name
                        org-export-registered-backends)))
    (dolist (backend val)
      (cond
       ((not (load (format "ox-%s" backend) t t))
        (message "Problems while trying to load export back-end `%s'"
                 backend))
       ((not (memq backend new-list)) (push backend new-list))))
    (set-default 'org-export-backends new-list)))

So to say we will not mess with it but you can see there is some complication in managing it. Thankfully, the manual tells us how to create a simple custom exporter according to this link which is simply: 

(defun orgtbl-to-language (table params)
  "Convert the orgtbl-mode TABLE to language."
  (orgtbl-to-generic
   table
   (org-combine-plists
    '(:tstart "!BTBL!" :tend "!ETBL!" :lstart "!BL!" :lend "!EL!" :sep "\t")
    params)))

After reading the documentation, we are interested in only two properties: :skip and :lfmt. The later which is more important controls how a line is formatted and it accepts a function which takes a list of values and returns the line it represents. Since this is YAML, our delimiter is a newline, the initial prefix of a dash and space, adding colons and quoting it for safety. The former just allows us to skip printing the header line since it is in the cell value like with JSON but unlike CSV.

The plan of attack is to simply use a functional zip with the headers and row values, format accordingly and profit. The header can be found with car of the table parameter which is not so obvious. So I shiv my own zip function and be done with it.

So we're done. That was quick.

Org-Export

Since I am using this with my org-jekyll-blogger.el I have to accommodate how to export this. Since I am using an Org-Table, it will be unintentionally exported to an html file since it is an org file. So I had to configure how files from the data will be exported but the weird thing is that Jekyll doesn't search data directories recursively like the posts directory but that's something easy to resolve but I wonder if it is a bug, unlikely though.

As a final point is how to set the export options for a table. One feature I can ask is to auto export then publish the table when it is saved just like my org-jekyll-blogger-auto-publish-on-save. And obviously I can shiv that on the spot. 

(defun fn/org-jekyll-blogger-export-and-publish ()
  "Export a table and publish the file accordingly."
  (interactive)
  (if (not (org-at-table-p))
      (if (not (called-interactively-p 'interactive))
          (message "No table at point to publish.")
        (error "Point is not at a table"))
    (org-table-export)

    (lexical-let ((export-file (org-entry-get (point) "TABLE_EXPORT_FILE" t)))
      (if (not export-file)
          (message "No TABLE_EXPORT_FILE property")
        (lexical-let ((export-buffer (find-file-noselect export-file)))
          (with-current-buffer export-buffer
            (org-publish-current-file))
          (message "Table published."))))))

(defun fn/org-jekyll-blogger-auto-publish-table-on-save ()
  "Auto export and publish table on save."
  (interactive)
  (add-hook 'after-save-hook #'fn/org-jekyll-blogger-export-and-publish t t))

So with this small snippet it will auto publish the table when it is saved assuming it has a containing org project. Not the best code but it does the job and not that hard to write although I had to peek at org-table-export to determine the name of the file. The oddity in this code is using called-interactively-p which merely indicates if the containing function is called as a command or a function; this allows me to either throw an error message or an info message depending on the context. For example, using it directly you should get an error that you need to mark what table you should export with the point; but if it is called by a hook, you wouldn't want it to throw an error since there might be other hooks in play so better an info message.
